drydock is a sandbox for autonomous coding agents on macOS. Each task runs in a per-task hardware-isolated VM; the agent never sees your real Anthropic key — a host-side credential gateway issues short-lived budgeted bearer tokens. Egress is deny-by-default. The only artifact that leaves the sandbox is a captured git diff, and it doesn't reach origin until you approve it.

01 Why

Assume the VM is already hostile.

The agent runs in bypass mode — every tool call executes with no confirmation. The threat isn't the agent misbehaving; it's the code it runs: a malicious dependency, a poisoned repository, a prompt-injection that turns a fetched URL into a shell command. drydock treats the sandbox as compromised by default and contains the blast radius with two boundaries that don't depend on the agent's good behavior: hardware isolation on one axis, deny-by-default egress on the other.

Exactly one thing crosses inward (a per-task token + the instruction) and exactly one crosses outward (a diff a human approves). The agent produces data, never a command the host executes on its behalf.

02 Architecture

One broker, two host proxies, a sealed VM.

Four flows cross the trust boundary, all labeled. The agent gets a bearer token (never the real key); the gateway swaps in X-Api-Key only on the host side; squid resolves and CONNECTs allowlisted hosts; everything else is dropped by the in-VM nft pin. The single artifact that crosses back is a captured git diff, and it doesn't reach origin without your approval.

03 Three guarantees

What the containment actually buys.

/ 01

The key is never in the VM

The sandbox holds only a per-task bearer token. An in-broker reverse proxy validates it, strips Authorization, and injects the real X-Api-Key. Leak the token: works only through your gateway, capped by budget, revoked at task end.

/ 02

Egress is deny-by-default

An in-VM nft pin allows exactly two host ports — gateway and proxy — and drops DNS. A host-side squid resolves names and allows registries by hostname. The allowlist lives in the host, not in the territory the attacker controls.

/ 03

No diff lands without you

.git is host-only; only the work tree is mounted. brokerd captures the diff, persists it, and blocks the original request until you run drydock approve. Planted hooks never execute on the host; auto_approve must be opted into explicitly.

04 Limits

What it deliberately doesn't defend.

Containments fail quietly when they overclaim. The full threat model is the contract; here are the residuals an operator should know about before installing.

Out of scopeWhy drydock can't stop it
Operator approves a malicious diffThe gate makes review possible, not automatic. A subtle backdoor in 200 lines of plausible code lands if you click through. Diff like you would for an external contributor.
Prompt injection from staged repo filesA hostile AGENTS.md, .cursorrules, or docstring inside the cloned repo can steer the agent for that task. drydock doesn't filter staged content — the diff gate is the backstop.
Side-channel exfil via the diff itselfA clever agent can encode bytes in file names, whitespace, or comment text. drydock makes this visible at review; it doesn't detect it programmatically.
Host compromiseMalware on the Mac running brokerd reads keys, mounts sockets, edits binaries. drydock assumes the host is trusted.
Guest-to-host escape in Apple containerA VM-stack 0day defeats every claim above. drydock pins the tested version and warns on drift; the escape itself is upstream.
Operator's GitHub/GitLab credentialsdrydock pushes via the host's gh / glab. Whoever can run drydock approve can push to anything those tokens can reach.
Cost exhaustion within the budgetThe USD cap stops runaway spend, not runaway uselessness. An agent that burns $2 on no-op calls hits the cap and returns no diff.
05 The credential path

A token goes in. A real key never comes back.

The gateway is the only component that ever holds the real key. It validates the bearer against an in-memory lease, swaps the credential, forwards over TLS, and meters response usage against the task's USD budget. Revocation is a single map delete; a broker crash invalidates every token for free.

gateway — request path
# inside the VM — all it ever holds
ANTHROPIC_BASE_URL = http://192.168.66.1:8088
ANTHROPIC_AUTH_TOKEN = tok_9f3c…a17b   # per-task, revocable

# at the gateway (host TCB) — never logged, never returned to the VM
validate(tok) → lease{ budget: $2.00, spent: $0.41 }
strip  Authorization: Bearer tok_…
set    X-Api-Key: sk-ant-…REAL
forward → https://api.anthropic.com   # real TLS, host→Anthropic
meter  usage → spent += cost(model, in, out)
# task ends → Revoke(tok) → 401 forever after
06 The git trap

A diff is data. .git is a weapon.

Mount a normal clone into the VM and then run git commit/push on the host against that same repo: a malicious agent can drop .git/hooks/pre-commit or poison .git/config — and those execute on the host, with the host's credentials. RCE straight through the trust boundary.

drydock clones on the host, moves .git out of the mounted path, mounts only the work tree, and runs host git with core.hooksPath=/dev/null. The diff — and nothing else — crosses back. Adversarially reviewed; regression-tested.

07 Status

Audited, race-clean, shipping today.

drydock has been adversarially reviewed (credential gateway, no blocking findings), boots cleanly on apple/container 1.0 + Apple silicon, and runs the full task lifecycle — POST, VM run, gated diff, approve, push — under unit, integration, and -race tests. Install via the Homebrew tap (brew install sricola/drydock/drydock); the current release is v0.1.2. Precise security claims live in the threat model.

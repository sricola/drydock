drydock runs Claude Code as an autonomous coding agent inside a per-task Linux VM on Apple silicon. The model key never enters the sandbox, egress is deny-by-default, and git push happens on the host — so --dangerously-skip-permissions is safe to run unattended.

01 The adversary

Assume the VM is already hostile.

The agent runs in bypass mode: every tool call — file writes, shell commands, web fetches, subagent spawns — executes with no confirmation. So the threat isn't the agent misbehaving; it's the code it runs: a malicious dependency, a poisoned repository, a prompt-injection that turns a fetched web page into a shell command.

drydock treats the sandbox as compromised by default and contains the blast radius with the VM boundary plus a tightly-scoped egress allowlist. The in-VM permission prompts become redundant friction once that containment is real — which is exactly what makes unattended, headless runs tractable.

02 The trust boundary

Commands never cross. Only data.

Two zones. Inside the VM: untrusted code. On the host: the trusted computing base — the broker, the credential gateway, the egress proxy, and the real git/model credentials. Exactly one thing crosses inward (a per-task token plus the task instruction), and exactly one thing crosses outward (a git diff).

That asymmetry is the whole design. The agent produces a diff — inert data the host inspects and applies — never a command the host executes on its behalf.

04 Three guarantees

What the containment actually buys.

/ 01

The key is never in the VM

The sandbox holds only a per-task bearer token. An in-broker reverse proxy validates it, strips Authorization, and injects the real X-Api-Key on the way out. Leak the token and it works only through your gateway, capped by budget, revoked the instant the task ends.

/ 02

Egress is deny-by-default

An in-VM nft pin allows traffic to exactly two host ports — gateway and proxy — and drops DNS. A host-side squid resolves names and allows registries by hostname. The allowlist lives in the trusted host, not in the territory the attacker controls.

/ 03

Push happens host-side

The broker clones on the host and keeps .git host-only — only the work tree is mounted. So a VM that plants .git/hooks can't execute code on the host at push time, and clone-URL credentials never reach the sandbox.

05 The credential path

A token goes in. A real key never comes back.

Claude Code is pointed at the gateway with ANTHROPIC_BASE_URL and authenticates with a bearer token in ANTHROPIC_AUTH_TOKEN. The gateway is the only component that ever holds the real key — it validates the token against an in-memory lease, swaps the credential, forwards over TLS, and meters the response usage against a per-task dollar budget. Revocation is a single map delete; a broker crash invalidates every token for free.

gateway — request path
# inside the VM — all it ever holds
ANTHROPIC_BASE_URL = http://192.168.66.1:8088
ANTHROPIC_AUTH_TOKEN = tok_9f3c…a17b   # per-task, revocable

# at the gateway (host TCB) — never logged, never returned to the VM
validate(tok) → lease{ budget: $2.00, spent: $0.41 }
strip  Authorization: Bearer tok_…
set    X-Api-Key: sk-ant-…REAL
forward → https://api.anthropic.com   # real TLS, host→Anthropic
meter  usage → spent += cost(model, in, out)
# task ends → Revoke(tok) → 401 forever after
06 The git boundary

A diff is data. .git is a weapon.

The subtle vector: if you mount a normal clone into the VM and then run git commit/push on the host against that same repo, a malicious agent can drop a .git/hooks/pre-commit or poison .git/config — and those execute on the host, with the host's credentials. That's remote code execution straight through the trust boundary.

drydock separates the two: clone on the host, move .git out of the mounted path, mount only the work tree, and run host git with core.hooksPath=/dev/null and fsmonitor off. The diff — and nothing else — crosses back. This was caught by an adversarial review and is regression-tested.

08 Footprint

A small, inspectable Go broker.

No heavy framework — a single daemon orchestrating Apple's container CLI, with focused packages you can hold in your head.

gateway · token proxy + budget netfw · squid allowlist + vmnet creds · grant / revoke seam stage · clone-in / diff-out runner · container argv broker · the orchestration loop

Verified on hardware. VM boot, the nft pin (direct internet + DNS fail closed), the gateway (VM → 401 on an invalid token), and squid (pypi allowed → 200, example denied → 403) are all confirmed on apple/container 1.0. The credential gateway passed an independent adversarial review with no blocking findings.
